# Redis Position Read Model

User position endpoints read a Redis projection, while PostgreSQL remains the sole authority for balances, positions, collateral, risk checks, liquidation, and audit queries. The detailed Chinese design and operating guide is [position-redis-cache_CN.md](position-redis-cache_CN.md).

The projection is keyed by product line and user and uses one Redis Cluster hash tag for position, collateral, and revision hashes. Account-provider captures one final snapshot per changed key and stores it as the complete durable position event in the same transaction. The event uses `<PRODUCT_LINE>:<userId>` as its Kafka key. After commit, that exact snapshot is offered to a bounded coalescing local worker for low latency; a dedicated account position-cache consumer replays the same event for durable recovery.

Redis is deliberately not part of an XA transaction. PostgreSQL is authoritative and the cache is a rebuildable, revisioned read model. A Lua compare-and-set makes local acceleration, Kafka replay, and PostgreSQL rebuild safe when they race. Queue overflow, malformed events, or Redis write failure removes the product-line readiness marker. The Kafka consumer retries without skipping committed financial events, while the coordinator remains the bounded reconciliation fallback. If the marker is absent or Redis fails, user reads fail closed with HTTP 503 rather than falling back to PostgreSQL.
